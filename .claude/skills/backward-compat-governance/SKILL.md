# 後方互換性ガバナンス

互換性を「残すコード」ではなく「契約と撤去計画」として管理する。

## 核心原則

後方互換性それ自体が悪ではない。互換性を支える「仕様・テスト・撤去計画・計測」が欠けたとき、
互換要求が実装へ流れ込みゴミコード化する。

## 対策パターン

1. **公開API境界の明確化**: 互換対象を限定
2. **非推奨化サイクル**: 宣言 → 猶予 → 削除
3. **互換層の局所化**: Adapter/Strangler Figパターン
4. **契約テスト**: Consumer-Driven Contract
5. **AI生成コードの互換性ゲート**

## アンチパターン検出

```
❌ if (version < X) { /* 旧挙動 */ } else { /* 新挙動 */ }
❌ @Deprecated が何年も残るAPI
❌ LegacyXxxAdapter, OldXxxWrapper の増殖
❌ "互換性のため" コメントの散在
```

## 関連スキル（併読推奨）
- `clean-architecture`: 互換層を配置するアダプタ層の設計
- `repository-design`: 公開API境界としてのリポジトリの互換性管理
- `intent-based-dedup`: 旧API・新APIの意図的な重複を共通化しない判断
